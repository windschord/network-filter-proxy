# US-004: Management REST API によるルール動的管理

## ユーザーストーリー

**私は** ClaudeWork オペレーターとして
**〜したい** プロキシを再起動せずにルールセットを動的に追加・変更・削除したい
**なぜなら** Claudeセッションの開始・終了のたびにプロキシを再起動するとダウンタイムが発生するから

## 受入基準（EARS記法）

### 機能要件

- **REQ-004-001**: システムは TCP ポート 8080 で Management REST API をリッスンしなければならない
- **REQ-004-002**: 環境変数 `API_PORT` が設定されている場合、システムは指定されたポートで API をリッスンしなければならない
- **REQ-004-003**: `GET /api/v1/rules` リクエストを受信した時、システムは全ての送信元 IP のルールセットを JSON 形式で返さなければならない
- **REQ-004-004**: `PUT /api/v1/rules/{sourceIP}` リクエストを受信した時、システムは指定した送信元 IP のルールセットを全置換し、即時（再起動なしに）反映しなければならない
- **REQ-004-005**: `PUT /api/v1/rules/{sourceIP}` で送信元 IP が未登録の場合、システムは新規ルールセットとして作成しなければならない
- **REQ-004-006**: `DELETE /api/v1/rules/{sourceIP}` リクエストを受信した時、システムは指定した送信元 IP のルールセットを削除し HTTP 204 を返さなければならない
- **REQ-004-007**: `DELETE /api/v1/rules/{sourceIP}` で指定した送信元 IP が存在しない場合、システムは HTTP 404 とエラーメッセージを返さなければならない
- **REQ-004-008**: `DELETE /api/v1/rules` リクエストを受信した時、システムは全ルールセットを削除し HTTP 204 を返さなければならない
- **REQ-004-009**: `PUT /api/v1/rules/{sourceIP}` のリクエストボディのバリデーションに失敗した場合、システムは HTTP 400 とバリデーションエラー詳細を JSON 形式で返さなければならない
- **REQ-004-010**: システムは Management API に認証機能を設けてはならない（internal network 内でのみ利用することを前提とする）

### バリデーション要件

- **REQ-004-011**: `host` フィールドが空文字の場合、システムはバリデーションエラーを返さなければならない
- **REQ-004-012**: `host` フィールドが `*.*.example.com` のような多段ワイルドカードの場合、システムはバリデーションエラーを返さなければならない
- **REQ-004-013**: `port` フィールドが 1〜65535 の範囲外の場合、システムはバリデーションエラーを返さなければならない

## データ要件

### API エンドポイント一覧

| メソッド | パス | 説明 | 成功レスポンス |
|---------|------|------|--------------|
| GET | `/api/v1/rules` | 全ルール一覧取得 | 200 |
| PUT | `/api/v1/rules/{sourceIP}` | ルールセット全置換 | 200 |
| DELETE | `/api/v1/rules/{sourceIP}` | 指定 IP のルール削除 | 204 |
| DELETE | `/api/v1/rules` | 全ルール削除 | 204 |

### PUT レスポンス 200 例

```json
{
  "source_ip": "172.20.0.3",
  "entries": [
    { "host": "api.anthropic.com", "port": 443 }
  ],
  "updated_at": "2026-03-09T10:30:00Z"
}
```

### PUT バリデーションエラー 400 例

```json
{
  "error": "validation_error",
  "message": "invalid host pattern: **.example.com",
  "details": [
    { "field": "entries[0].host", "message": "invalid wildcard pattern" }
  ]
}
```

### DELETE 404 例

```json
{
  "error": "not_found",
  "message": "no rules found for source IP: 172.20.0.99"
}
```

## 備考

- ルールの変更は即時反映（メモリ上のマップを同期的に更新）
- 並行アクセスに対してスレッドセーフな実装が必要（`sync.RWMutex` 等）
- API ポートはホスト側の external network に露出しない運用を想定
