# US-002: ドメインベースのホワイトリスト制御

## ユーザーストーリー

**私は** ClaudeWork オペレーターとして
**〜したい** 許可リストに登録されたドメイン・IP のみへの通信を許可したい
**なぜなら** ClaudeCode コンテナが意図しない外部サービスにアクセスすることを防ぐ必要があるから

## 受入基準（EARS記法）

### 機能要件

- **REQ-002-001**: CONNECT リクエストを受信した時、システムは宛先ホスト名とポートがルールセット内のいずれかのエントリにマッチするか判定しなければならない
- **REQ-002-002**: もしルールにマッチするエントリが存在する場合、システムは接続を許可し HTTP 200 を返さなければならない
- **REQ-002-003**: もしルールにマッチするエントリが存在しない場合、システムは `403 Forbidden` を返しレスポンスヘッダーに `X-Filter-Reason: denied` を付与しなければならない
- **REQ-002-004**: ルールエントリのホストが完全一致ドメインの場合、システムは宛先ホストが完全に一致する場合のみ許可しなければならない
- **REQ-002-005**: ルールエントリのホストが `*.example.com` 形式のワイルドカードの場合、システムは `example.com` 自体および直下1階層のサブドメイン（`foo.example.com` 等）にのみマッチしなければならない。`evil.foo.example.com` のような多段サブドメインはマッチしない
- **REQ-002-006**: ワイルドカードは先頭の 1 セグメントのみサポートし、`*.*.example.com` のような多段ワイルドカードは無効パターンとして拒否しなければならない
- **REQ-002-007**: ルールエントリのホストが IPv4 アドレスの場合、システムは宛先 IP が完全一致する場合のみ許可しなければならない
- **REQ-002-008**: ルールエントリのホストが CIDR 表記（例: `140.82.112.0/20`）の場合、システムは宛先 IP が CIDR 範囲内に含まれる場合に許可しなければならない
- **REQ-002-009**: ルールエントリに `port` が指定されている場合、システムは宛先ポートが一致する場合のみ許可しなければならない
- **REQ-002-010**: ルールエントリに `port` が指定されていない場合、システムは全ポートを許可しなければならない

## データ要件

### ルールエントリ構造

| フィールド | 型 | 必須 | 説明 |
|-----------|------|------|------|
| `host` | string | Yes | ドメイン、ワイルドカード、IP アドレス、CIDR |
| `port` | int | No | 1〜65535。省略時は全ポート許可 |

### ホストマッチングルール

| パターン例 | マッチ対象 |
|-----------|-----------|
| `api.anthropic.com` | `api.anthropic.com` のみ |
| `*.github.com` | `github.com`, `api.github.com`, `raw.github.com` 等（直下1階層のみ。`evil.api.github.com` は不可） |
| `140.82.112.3` | IP `140.82.112.3` への接続 |
| `140.82.112.0/20` | CIDR 範囲内の全 IP |
