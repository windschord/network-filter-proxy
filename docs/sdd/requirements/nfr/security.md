# セキュリティ要件

## 概要

プロキシはネットワーク通信の制御ゲートウェイとして機能するため、バイパス攻撃や誤設定による情報漏洩を防ぐことを最優先とする。

## 要件一覧

### NFR-SEC-001: デフォルト拒否原則

**EARS記法**: システムは登録されていない送信元 IP からの全ての通信を拒否しなければならない

**測定基準**:
- 指標: 未登録 IP からの通信拒否率
- 目標値: 100%（例外なし）
- 測定方法: 未登録 IP からのアクセステスト

**関連する機能要件**: REQ-003-003

**優先度**: 高

**検証方法**:
- [ ] 未登録 IP から CONNECT リクエストを送り 403 + `X-Filter-Reason: no-rules` が返ることを確認

---

### NFR-SEC-002: TLS 非終端

**EARS記法**: システムは CONNECT トンネル確立後、クライアントとサーバー間の TLS 通信を傍受・復号してはならない

**測定基準**:
- 指標: TLS セッションの完全性
- 目標値: プロキシによる証明書の発行・置換が行われないこと
- 測定方法: コードレビューおよび通信キャプチャ

**関連する機能要件**: REQ-001-002

**優先度**: 高

**検証方法**:
- [ ] プロキシ経由の HTTPS 通信でサーバー証明書が変化しないことを確認

---

### NFR-SEC-003: Management API アクセス制限

**EARS記法**: Management API（:8080）はデフォルトで `127.0.0.1` にバインドしなければならない。`API_BIND_ADDR` で `0.0.0.0` 等に変更する場合、外部ネットワークへの露出はインフラ側のネットワーク分離（Docker Compose の internal network 等）で防止しなければならない

**測定基準**:
- 指標: 外部ネットワークからの API アクセス可否
- 目標値: 外部からアクセス不可
- 測定方法: ネットワーク設定のレビュー

**関連する機能要件**: REQ-004-010, REQ-008-001, REQ-008-002

**優先度**: 高

**検証方法**:
- [ ] デフォルト（`API_BIND_ADDR` 未設定）で `127.0.0.1` にバインドされることを確認
- [ ] ネットワーク設定（Docker Compose / Kubernetes NetworkPolicy 等）で API ポートが外部ネットワークから到達不可能であることを確認

---

### NFR-SEC-004: ルールバイパス防止

**EARS記法**: システムはホスト名の正規化（大文字小文字の統一等）を行い、ルール迂回を防止しなければならない

**測定基準**:
- 指標: 大文字小文字の差異・末尾ドット等によるバイパス成功率
- 目標値: 0%
- 測定方法: 境界値テスト

**関連する機能要件**: REQ-002-001

**優先度**: 中

**検証方法**:
- [ ] `API.ANTHROPIC.COM` や `api.anthropic.com.` 等の大文字・末尾ドット変形でバイパスできないことを確認
- [ ] IDN/Punycode 表記（例: `xn--e1afmapc.example.com`）を含むホスト名でルールバイパスが発生しないことを確認
- [ ] IPv6 アドレス正規化（例: `::1` と `0:0:0:0:0:0:0:1` の同一視）でバイパスが発生しないことを確認
- [ ] 連続ドット（例: `api..github.com`）が ValidateEntry でエラーとなり登録・マッチングに使用されないことを確認

---

### NFR-SEC-005: 並行アクセスの安全性

**EARS記法**: 複数のリクエストが同時にルールセットを参照・変更している間、システムはデータ競合を発生させてはならない

**測定基準**:
- 指標: データ競合（race condition）の発生
- 目標値: 0 件
- 測定方法: `go test -race` による競合検出

**関連する機能要件**: REQ-004-004

**優先度**: 高

**検証方法**:
- [ ] `go test -race ./...` が全テストでパスすること

---

### NFR-SEC-006: PII ログ保護

**EARS記法**: システムはログに個人識別情報（PII）を出力してはならない。ログに含めるのは送信元 IP・宛先ホスト・ポート・判定結果のみとし、リクエストボディや認証情報を含めてはならない

**測定基準**:
- 指標: ログ出力内容の PII 含有
- 目標値: 0 件
- 測定方法: コードレビューおよびログ出力サンプリング

**関連する機能要件**: REQ-006-001〜003

**優先度**: 中

**検証方法**:
- [ ] ログ出力フィールドが `timestamp`、`level`、`action`、`src_ip`、`dst_host`、`dst_port`、`reason` のみであることを確認

---

### NFR-SEC-007: DoS 保護

**EARS記法**: システムは単一クライアントからの大量リクエストによってサービス全体が停止しないよう、接続数・処理時間の観点で合理的な制限を設けなければならない

**測定基準**:
- 指標: 単一クライアントによるサービス停止の可否
- 目標値: サービス停止なし
- 測定方法: 負荷テストによる検証

**関連する機能要件**: REQ-001-001

**優先度**: 中

**検証方法**:
- [ ] 100 同時接続下でも他クライアントのリクエストが処理されることを確認（NFR-PERF-002 と共通）

---

### NFR-SEC-008: 監査ログ

**EARS記法**: システムはルール変更操作（PUT / DELETE）をすべて監査ログとして記録しなければならない。操作種別・対象 IP・変更後エントリ数・タイムスタンプを含めること

**測定基準**:
- 指標: ルール変更操作のログ記録率
- 目標値: 100%（漏れなし）
- 測定方法: 全 CRUD エンドポイントのログ出力テスト

**関連する機能要件**: REQ-006-003

**優先度**: 中

**検証方法**:
- [ ] PUT / DELETE 各エンドポイント呼び出し時に `operation`、`src_ip`（または `all`）、`count`、`timestamp` を含むログが出力されることを確認
