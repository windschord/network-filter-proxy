# DEC-005: Go バイナリの healthcheck サブコマンドで Docker ヘルスチェックを実現

## ステータス

**ステータス**: 承認済
**決定日**: 2026-03-12

## コンテキスト

Docker Compose の `depends_on: condition: service_healthy` でプロキシコンテナの起動完了を待機する必要がある。ベースイメージが `gcr.io/distroless/static:nonroot` のため `curl`/`wget` が利用できない。

### 現状の課題
- distroless イメージに HTTP クライアントツールがない
- 従来の設計では Dockerfile 内の HEALTHCHECK を「不要」としていたが、Docker Compose 環境での起動順序制御に必要と判明

---

## 検討した選択肢

### 選択肢A: Go バイナリに healthcheck サブコマンドを追加（採用）

**概要**: `filter-proxy healthcheck` で `127.0.0.1:{API_PORT}/api/v1/health` にリクエストし、結果に応じて終了コードを返す。

**メリット**:
- 追加のバイナリや依存が不要
- distroless イメージとの相性が良い
- 実装がシンプル（10-15 行程度）

**デメリット**:
- main.go にサブコマンド分岐が入る

### 選択肢B: 別の軽量ヘルスチェックバイナリを同梱

**メリット**:
- main バイナリの責務が増えない

**デメリット**:
- ビルド対象が増える
- Docker イメージのサイズが増加
- 過剰設計

---

## 決定

**選択**: 選択肢A（Go バイナリに healthcheck サブコマンドを追加）

**根拠**: 最もシンプルで、Issue #6 の要望とも一致。distroless イメージで curl が使えない問題を Go バイナリ自身で解決する標準的なパターン。
